# Paper-figure reproducibility corpus

This directory contains the **minimum bbsm / bbky / verify artifacts needed to
reproduce every numerical claim, table row, and figure data point** in
`01_arxiv_full_technical_report/bbsolver_arxiv_full_technical_report_v3.md`.

It is a subset of the development corpus that lives at
`bbsolver/artifacts/bbsolver/corpus/live_runs/` (and historically at
`(alternative)/artifacts/(alternative)/corpus/live_runs/`). The full corpus is roughly
1 GB and includes development iterations, ablation runs, and harness cleanup
intermediates that the paper does not reference. This pruned bundle is
~62 MB and ships every request_id cited by the supplementary CSVs in
`../supplementary/`.

## Layout

```
paper_corpus/
├── corpus_manifest.csv            # request_id → paper section + fixture + ε + stats
├── README.md                      # this file
└── req-<epoch_ms>/                # one folder per cited solve
    ├── <req>_g<n>.bbsm.json       # SampleBundle input (one per property group)
    ├── <req>_g<n>.bbky.json       # solved KeyBundle output
    ├── <req>_g<n>.bbky.json.progress.log
    │                              # per-property progress events (sample,
    │                              # segment, refit, etc.) — source of all
    │                              # solve_time_ms numbers in §Operating modes
    ├── <req>.verify.json          # post-solve verify pass result
    │                              # (per-property max_err vs source samples)
    └── last_verify_card.txt       # human-readable verify summary
```

The `_cleanup_*` intermediate files generated by the AE harness during its
two-pass cleanup workflow are **not** included here — they are debugging
artifacts that do not back any paper claim. The `progress.log` files contain
`solve_time_ms` and per-property phase timing used in the §Operating modes
cost table.

## Reproducing a paper row

Each bbsm / bbky pair can be re-solved or re-verified with the standalone
`bbsolver` binary from the public source release at
https://github.com/ivg-design/bbsolver:

```sh
# Build bbsolver (one-time, from repo root)
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build -j

# Re-solve a paper row from its bbsm input
./build/bbsolver solve \
    benchmarks/corpus/req-1779727765498/req-1779727765498_g1.bbsm.json \
    /tmp/reproduce_g1.bbky.json \
    --tolerance 0.05 --screen-px 1 --jobs 0

# Compare against the shipped bbky
diff -q /tmp/reproduce_g1.bbky.json \
        benchmarks/corpus/req-1779727765498/req-1779727765498_g1.bbky.json

# Re-verify against source samples
./build/bbsolver verify \
    benchmarks/corpus/req-1779727765498/req-1779727765498_g1.bbky.json \
    benchmarks/corpus/req-1779727765498/req-1779727765498_g1.bbsm.json
```

The published `solve_time_ms` is hardware-dependent; the per-property
`max_err` values are deterministic and should match the shipped `verify.json`
bit-for-bit on any platform.

## Mapping back to the paper

See `corpus_manifest.csv`. The columns are:

- `request_id` — folder name in this corpus
- `paper_section` — exact table or figure citation
- `fixture` — short description of the rig / scene
- `eps` — solver `--tolerance` parameter
- `n_properties`, `n_samples_input`, `n_keys_output` — for sanity-checking
- `combined_compression_ratio` — headline number reported in the paper
- `description` — one-line rationale for why this request is in the paper

For per-row Pareto data and per-property residual breakdowns, see the CSVs
under `../supplementary/`.

## Reproducing the After Effects round-trip

The bbsm input files capture the sampled state of the AE composition at
solve time — they can drive a standalone bbsolver run without needing AE
installed. To reproduce the **host round-trip** (sample → solve → writeback
→ re-evaluate AE playback → verify), you also need:

- After Effects 2024+ on macOS or Windows
- The AE benchmark project at
  `../../after_effects_benchmark_project/bbSolver_benchmarking.aep`,
  which contains the DUIK humanoid rig, the ant rig, the FK noodle setup,
  and the variable-topology blob compositions used by §5.2 and §5.7
- The `bbsolver-test-harness` ScriptUI panel from
  https://github.com/ivg-design/bbsolver `examples/after-effects/`

The harness will produce a new `req-<epoch_ms>` folder under
`bbsolver/artifacts/bbsolver/corpus/live_runs/` for each solve; you can then
diff against the corresponding folder in this corpus.

## Provenance

The raw solver inputs and outputs — `<req>_g<n>.bbsm.json`,
`<req>_g<n>.bbky.json`, and `<req>_g<n>.bbky.json.progress.log` — are
byte-identical copies of the original live_runs artifacts produced during
paper preparation. Their `max_err` and key-count values have not been
recomputed or edited.

The post-solve verification artifacts — `<req>.verify.json` and
`last_verify_card.txt` — were regenerated by re-running the canonical
`bbsolver verify` CLI (v1.0.1) against each `bbky.json + bbsm.json` pair
so the entire corpus shares one verifier provenance. v1.0.1 is a verifier
bug fix (variable-topology `shape_flat` strict-dim check); the solver
itself is unchanged from v1.0.0, so running v1.0.1 against the shipped
samples produces bit-identical keys. Each regenerated `verify.json`
carries a top-level `regenerator: "bbsolver verify (canonical CLI)"`
field for explicit provenance. See `THRESHOLD_NOTE.md` for the full
diagnosis.

Dates of the underlying runs are encoded in the request_id (epoch ms
since 1970-01-01); all bundles were produced between 2026-05-25 and
2026-05-26 (solver), and re-verified on 2026-05-27 (verifier).
